Core Web Vitals no Shopify: LCP, CLS e INP

Performance 12 min leitura Atualizado em Mar/2026

Atingir 90+ no Lighthouse em uma loja Shopify real — com apps, pixels de rastreamento e carrossel de imagens — é difícil. Não porque o Shopify seja lento, mas porque cada app instalado adiciona JavaScript ao thread principal, cada imagem não otimizada aumenta o LCP, e cada elemento sem dimensão reservada empurra o layout durante o carregamento.

Neste guia documentamos as técnicas que aplicamos em produção para controlar os três Core Web Vitals que o Google mede: LCP, CLS e INP.

1. LCP, CLS e INP: o que o Google mede

O Google PageSpeed Insights e o Chrome User Experience Report (CrUX) avaliam três dimensões distintas de experiência:

  • LCP — Largest Contentful Paint: tempo até o maior elemento visível (hero banner ou imagem de produto) ser renderizado. Meta: abaixo de 2,5 segundos. Acima de 4s é considerado "Ruim".
  • CLS — Cumulative Layout Shift: soma de todos os deslocamentos visuais inesperados durante o carregamento. Meta: abaixo de 0,1. Um popup que empurra o conteúdo ou uma fonte que substitui uma fallback são causas clássicas.
  • INP — Interaction to Next Paint: mede o tempo entre qualquer interação do usuário (clique, tap, tecla) e a próxima atualização visual. Substituiu o FID em março de 2024. Meta: abaixo de 200ms.

Para e-commerce, o impacto é direto: cada segundo adicional de LCP reduz conversões em aproximadamente 12% (fonte: Google/Deloitte Digital, 2019 — ainda a referência mais citada na literatura de CRO). O CLS afeta a taxa de erro em formulários e cliques em botões errados em mobile. O INP ruim faz o botão "Adicionar ao Carrinho" parecer travado.

2. Otimizando o LCP no Shopify

O elemento LCP em 90% das lojas Shopify é o hero banner da home ou a imagem principal da página de produto. O caminho para um LCP rápido começa antes de o browser fazer o primeiro request.

Preload + fetchpriority no hero

O browser descobre a imagem LCP somente ao parsear o HTML e encontrar a tag <img>. Com preload, você instrui o browser a iniciar o download imediatamente, antes mesmo de processar o CSS ou JS que renderizaria esse elemento.

<!-- No <head> do seu theme.liquid --> <link rel="preload" as="image" href="{{ section.settings.hero_image | image_url: width: 1200 }}" fetchpriority="high" > <!-- Na tag img do hero --> <img src="{{ section.settings.hero_image | image_url: width: 1200 }}" srcset="{{ section.settings.hero_image | image_url: width: 600 }} 600w, {{ section.settings.hero_image | image_url: width: 900 }} 900w, {{ section.settings.hero_image | image_url: width: 1200 }} 1200w" sizes="100vw" fetchpriority="high" decoding="async" alt="{{ section.settings.hero_image.alt | escape }}" width="1200" height="600" />

Regra de ouro

Use fetchpriority="high" em apenas uma imagem por página — a que é o elemento LCP. Aplique loading="lazy" em todas as demais. Se você marcar duas imagens como high priority, o efeito é cancelado.

Formato WebP via Shopify CDN

O Shopify CDN (Fastly) suporta conversão de imagem em tempo real. Você não precisa subir arquivos WebP separados: basta usar o filtro image_url com o parâmetro correto.

{{ product.featured_image | image_url: width: 800, format: 'webp' }}

Imagens em WebP são tipicamente 25–35% menores do que JPEGs equivalentes. Para banners de 1–2 MB em JPG, isso significa 250–700 KB a menos por pageview.

3. Zerando o CLS no Shopify

Layout shift acontece quando o browser reserva zero espaço para um elemento e depois o renderiza. As três causas mais frequentes em Shopify:

Imagens sem dimensões explícitas

Toda <img> deve ter width e height definidos no HTML, ou aspect-ratio no CSS. Sem isso, o browser não sabe quanto espaço reservar antes de baixar a imagem.

/* CSS — reservar espaço antes do carregamento */ .product-card__image-wrapper { aspect-ratio: 1 / 1; /* quadrado — ajuste para o ratio das suas imagens */ overflow: hidden; background: #f5f5f5; /* placeholder visual enquanto carrega */ } .product-card__image-wrapper img { width: 100%; height: 100%; object-fit: cover; }

Fontes customizadas causando FOUT

Quando uma fonte customizada (Google Fonts, Adobe Fonts) não carregou ainda, o browser exibe o texto com a fonte fallback. Quando a fonte real chega, o layout inteiro reflui. Isso é chamado FOUT (Flash of Unstyled Text) e contribui para o CLS.

A solução é font-display: swap na declaração da fonte — o browser exibe o texto com fallback imediatamente e faz a troca suave, sem refluxo de layout:

/* No seu CSS de fontes */ @font-face { font-family: 'MinhaFonte'; src: url('minha-fonte.woff2') format('woff2'); font-display: swap; /* evita FOIT e reduz impacto no CLS */ }

Apps que injetam elementos dinâmicos

Popups de exit-intent, chat widgets (Tidio, JivoChat), e banners de cookie que aparecem depois que a página carregou são causas recorrentes de CLS. A solução de engenharia é reservar o espaço antes via CSS, mesmo quando o elemento ainda não existe no DOM:

/* Reservar espaço para o banner de cookie no rodapé */ body { padding-bottom: 80px; /* altura do banner */ } .cookie-banner { position: fixed; bottom: 0; left: 0; right: 0; height: 80px; }

4. Melhorando o INP no Shopify

INP ruim significa que clicar em "Adicionar ao Carrinho" ou em um filtro de categoria parece travado — o visual não responde imediatamente. O culpado quase sempre é JavaScript pesado no thread principal.

Long Tasks: o inimigo do INP

O Chrome define uma "Long Task" como qualquer tarefa JavaScript que ocupa o thread principal por mais de 50ms. Durante uma Long Task, o browser não pode processar interações do usuário. Para diagnosticar, abra o Chrome DevTools, vá em Performance, grave uma interação e procure barras vermelhas no thread principal.

Apps de terceiros são os maiores causadores. A abordagem é carregar scripts não críticos somente após o primeiro carregamento da página:

// Carregar apps não críticos após interação ou idle if ('requestIdleCallback' in window) { requestIdleCallback(() => { // carrega script de chat, popup, etc. const script = document.createElement('script'); script.src = 'https://cdn.app-terceiro.com/widget.js'; document.head.appendChild(script); }); } else { window.addEventListener('load', () => { // fallback para browsers sem requestIdleCallback setTimeout(() => { const script = document.createElement('script'); script.src = 'https://cdn.app-terceiro.com/widget.js'; document.head.appendChild(script); }, 3000); }); }

Diagnóstico com Chrome DevTools

No DevTools, aba Performance Insights, o campo "INP" mostra qual interação específica está lenta e qual script é responsável. Isso elimina o trabalho de adivinhar qual app está causando o problema.

5. Scripts de Terceiros: o maior obstáculo

Cada app Shopify instalado pode adicionar entre 50ms e 300ms ao carregamento. A regra que aplicamos em todos os projetos: nenhum script de app deve bloquear o first paint.

O atributo defer garante que o script só execute depois que o HTML foi completamente parseado. O async faz o download em paralelo mas pode executar antes do parse concluir, o que às vezes causa problemas de dependência:

<!-- Errado: bloqueia o render --> <script src="https://cdn.app.com/widget.js"></script> <!-- Correto: não bloqueia o parse do HTML --> <script src="https://cdn.app.com/widget.js" defer></script> <!-- Para scripts sem dependência de DOM --> <script src="https://cdn.app.com/analytics.js" async></script>

Para apps de baixa prioridade (chat, exit popup, reviews que aparecem abaixo do fold), vá além do defer e use o padrão de carregamento por IntersectionObserver: o script só é carregado quando o usuário scrolla até o elemento relevante.

6. Como auditar sua loja Shopify no Lighthouse

O Lighthouse mede desempenho simulado (CPU/rede throttled), não desempenho real. Os dados reais de usuários ficam no CrUX Report, acessível via PageSpeed Insights ou pelo Google Search Console em "Experiência na página". Use os dois:

  • Lighthouse (lab data): diagnóstico rápido de problemas específicos — ideal durante desenvolvimento e antes de cada deploy.
  • CrUX / Search Console (field data): dados reais agregados de usuários dos últimos 28 dias — é o que o Google usa no ranking. Um bom Lighthouse não garante bom CrUX se a base de usuários usa devices lentos.

Para testar em condições reais de produção Shopify, use o Chrome DevTools com throttling de CPU 4x. Isso simula um Moto G4, o dispositivo de referência do Lighthouse para mobile.

Chrome Coverage Tool

No DevTools, pressione Ctrl+Shift+P e pesquise "Coverage". A aba Coverage mostra quanto do JS/CSS carregado em cada app é efetivamente usado na página. Qualquer app com menos de 30% de uso é candidato a remoção ou carregamento lazy.

7. Resultados práticos que observamos

Em projetos de otimização de performance aplicando as técnicas deste guia, os padrões de melhoria que observamos com frequência são:

  • LCP reduzido de 4–6s para 1,8–2,4s com preload + WebP + remoção de render-blocking scripts
  • CLS zerado ou próximo de zero após adicionar aspect-ratio em cards de produto e font-display: swap nas fontes
  • INP melhorado de 400–600ms para abaixo de 200ms após diferir scripts de apps não críticos
  • Score Lighthouse mobile passando de 45–60 para 80–92 após otimização completa

O ponto de atenção: um score alto no Lighthouse de laboratório não equivale automaticamente a um bom CrUX. Se a base de clientes da loja usa muitos dispositivos de entrada (processador lento, memória limitada), o INP real medido pelo Google pode ser pior do que o simulado. O diagnóstico precisa sempre cruzar os dois.

Perguntas Frequentes

O que são Core Web Vitals e por que impactam uma loja Shopify?

Core Web Vitals são três métricas de experiência do usuário definidas pelo Google: LCP (velocidade de carregamento do conteúdo principal), CLS (estabilidade visual) e INP (velocidade de resposta a interações). Para lojas Shopify, essas métricas afetam diretamente o ranking orgânico, o Quality Score de anúncios pagos e a taxa de conversão.

Como melhorar o LCP de uma loja Shopify?

As três ações de maior impacto: (1) usar fetchpriority="high" na primeira imagem do hero banner junto com um preload no <head>; (2) converter imagens para WebP via filtro Liquid; (3) identificar e diferir scripts que bloqueiam o first paint, especialmente apps não críticos para o above-the-fold.

O que causa CLS alto no Shopify?

Causas mais comuns: imagens sem width/height ou sem aspect-ratio no CSS, fontes sem font-display: swap, apps de popup que injetam elementos no DOM após o carregamento. A solução primária é reservar espaço para todos os elementos antes que eles carreguem.

É possível atingir 90+ no Lighthouse com apps instalados?

Sim, mas exige auditoria de cada app com o Chrome DevTools Coverage. A abordagem é diferir todos os scripts não críticos para depois do first contentful paint, substituir apps pesados por implementações nativas em Liquid quando possível, e carregar via IntersectionObserver os widgets que aparecem abaixo do fold.

Sua loja está deixando conversões na mesa por lentidão?

Fazemos auditoria completa de Core Web Vitals com diagnóstico de LCP, CLS e INP e plano de otimização priorizado.
